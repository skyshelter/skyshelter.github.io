EVENT

BIG Picture

Event analysis

Defines a system by defining:

  • Events which a system should respond to for each event:
  • Incoming message(event flow or data flow)
  • Desired response
  • Actions or behaviors required to generate that response

Event to system(lots of actions) to Response

Event-driven systems

Eevent-driven systems are defined through the perspective of stimulus-response

  1. The system dose nothing until it is trigged by an event
  2. When an event occurs, the system responds as completely as possible
  3. After the response is complete, the system waits until another event occurs.

Type of event

  • External

    • External stimuli that a system responds to

    • A trigger that initiates the start of a process: some thing happens, that system should do something to responds

    • Often, a change in the system’s environment(someone or something has done something)

    • From outside the system

    • Initiated by external actor

  • Temporal

    • As a result of reaching a point in a time
    • Based on system deadlines
      • end of week, end of day, due time

  • State

    • Something inside the system triggers a processing need

Exmaple of these:

​ External:

  • Custormer contacts sales representative to order goods
  • User requests to view a picture
  • seller posts textbook details for sale

Temporal

  • Report is generated at the end of the month
  • Driver’s license expiry date is reached

State

  • Account balance is less than $0
    • can be outcome of other events

Event/response list

  1. list all stimuli[^刺激] (events) to which the system must respond

  2. For each stimulus, document the inputs provided and the response required

  3. For each stimulus-response pair, document the conditions of the system under which they should occur(or should not occur)

    uses:

    • identify requirements that need clarification
    • Discovering possible missing requirements
    • Communicate with the customer/user/client
    • Begin definition of the necessary requirements

How big is an event?

Remember how event-driven systems are defined:

  1. The system does nothing until it is trigged by an event
  2. When an event occurs, the system response as completely as possible
  3. After the response is complete, the system waits until another event occurs.

External event checklist

  • External actor wants something, resulting in a transaction
  • External actor wants some information
  • External data has changed and needs to be updated
  • Management wants information

Temporal event checklist

  • Internal or external outputs needed at specific points in time
  • Operational reports and documents
  • Transaction deadlines
  • Expiry dates

State event checklist

A system value reaches a particular threshold[^门槛].

  • Sometimes, an external event causes a system value to change which then should cause a state event to occur
    • eg. the account balance is 0.
  • Sometime similar to temporal events except that the point in time cannot be defined.

Other checklist

  • Are all stakeholders that interact with the system included?

    ==which means should analysis and list all stakeholders before find out the events==

  • Are all related event included ?

    • If there is a create, then should there also be read, update, and/or delete event?

      For instance, [read info], [updated info], [delete info], which means theses situation should be considered!!!

  • Are all events required for a transaction included?

  • Some event may actually be input data, conditions, or responses

  • Some event may not be relevant during analysis

An action by any other name

Also know as activities, tasks, processes, elementary business process
What work is performed to convert input(event) to outputs(response)

Minimal[^最小] model

In event-action-response analysis, actions are limited to the essential requirements

  • Essentia[^基本的] activity
    • Tasks the system needs to perform, assuming perfect technology
  • Essential memory
    • Data the system needs to remember to perform the essential activities
  • Minimal essential model
    • Everything you need, nothing you don’t

Beware of :

  • Requirements Creep

    • cool function
    • Future-proofing

  • Early design

    • We could build in this way
    • These decisions don’t belong in the analysis phase

  • Early implementation decisions

    • We could use this technology
    • Even worse than early design
      • (unfortunately, this happens a lot)

Event-action rules

  • one event
  • An action shouldn’t do different things for two different events
    • Two actions might do similar things
      • e.g. withdraw and deposit events both result in the account balance changing - but these two event result in two different actions, not a generic ‘ change account balance’ action)
  • Idleness check
    • When the action is complete, it does not do anything else until another event triggers it again.

Event data(input data)

​ All the data associated with the event is required by the activity.
​ No unused data

Response data(output data)

​ All the data associated with the response must be generated from the event and stored data
​ No miracle data

System output

Responses are system output
So they generally go to an actor, or an external entity.

If the system needs to do further processing, do not send a response back to the system!
Instead: Do it as part of the action
Do it as a state or temporal event

Naming events, actions and responses

Use the language of the domain, which means that the business and non-technical business people would use

Clear(unambiguous), Specific, Appropriate level of detail for the purpose of the model

e.g.

  • Do stuff

    what dose ‘do’ mean?
    waht dose ‘stuff’ mean?

  • Manage account
    What does ‘manage’ mean?

  • change email address
    ==This is an implementation[^执行] detail. Is the important event/action to ‘update contact details’?==
    “change email address” can be considered as an input which be change in the update contact details.

Grouping events

For a non-trivial system there are probably lost of event

Can group related events together

  • By sub-system
  • By related components or work items
  • By actor
  • By function(traditional modeling)
  • By object(object-oriented modeling)
